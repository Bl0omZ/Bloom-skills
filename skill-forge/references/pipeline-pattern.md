# Pipeline 模式

## 核心思想

Pipeline 强制执行有序的多步骤工作流，每步有明确的检查点。
不允许跳步，每步完成后验证输出，再进入下一步。
适合不可逆操作、高质量要求、或需要审计追踪的任务。

```
Step 1: [动作]
    │
    ▼ 检查点：[完成条件]
    │
    ├── 通过 → Step 2
    └── 失败 → 修复后重试 Step 1

Step 2: [动作]
    │
    ▼ 检查点：[完成条件]
    │
    └── ...
```

## SKILL.md 模板

```markdown

  触发关键词：[workflow]、发布、部署、流程、pipeline。
  注意：步骤必须按序执行，不可跳过。

# [Workflow] Pipeline

- [ ] [条件2]

若任意条件未满足：停止，报告未满足的条件，等待人工处理。

---

## Step 2：[步骤名称]

[步骤的具体指令]

完成条件：
- [ ] [条件1]
- [ ] [条件2]

---

## Step N：[最后步骤]

[步骤指令]

完成条件：
- [ ] [最终验证]

## Pipeline 完成报告

所有步骤完成后输出：

  Pipeline 执行完成
  - Step 1 [步骤名]：✅ 完成
  - Step 2 [步骤名]：✅ 完成
  - Step N [步骤名]：✅ 完成
  总耗时：[时间]
  输出物：[列表]

## 错误处理

任意步骤失败时：
1. 立即停止，不继续执行后续步骤
2. 报告：失败步骤、失败原因、当前状态
3. 提供恢复选项：A) 修复后从失败步骤重试  B) 从头重新执行  C) 中止
```

## 检查点设计原则

### 好的检查点 vs 坏的检查点

```
❌ 坏：检查步骤是否完成。（主观，无法验证）
✅ 好：输出文件存在且字数 > 500。（客观，可验证）

❌ 坏：确保质量良好。（无标准）
✅ 好：评分 >= 8/10 且无 CRITICAL 问题。（有明确阈值）

❌ 坏：用户满意。（无法自动化）
✅ 好：所有必填字段已填充，无 [TODO] 占位符残留。（可程序化检查）
```

### 检查点类型

| 类型 | 示例 | 适用场景 |
|------|------|----------|
| 存在性检查 | 「输出文件已创建」 | 文件生成步骤 |
| 完整性检查 | 「所有必填字段非空」 | 数据填充步骤 |
| 质量检查 | 「评分 >= 阈值」 | 内容生成步骤 |
| 人工确认 | 「用户输入 '确认'」 | 不可逆操作前 |
| 外部验证 | 「API 返回 200」 | 集成步骤 |

## 步骤数量建议

- 3-7 步：最佳（清晰且不繁琐）
- < 3 步：考虑用 Generator 替代
- > 7 步：拆分为子 Pipeline 或使用 `references/` 存放详细步骤

## 与其他模式组合

- **Pipeline + Reviewer**：Pipeline 最后一步用 Reviewer 模式做质量检查
- **Inversion + Pipeline**：先用 Inversion 收集配置，再用 Pipeline 执行
- **Pipeline + Generator**：Pipeline 某步骤调用 Generator 生成文档

## 典型使用场景

- 软件发布流程（测试 → 构建 → 部署 → 验证）
- 文档生成流水线（大纲 → 初稿 → 审查 → 定稿）
- 安全审计流程（侦察 → 分析 → 报告 → 复查）
- 数据处理管道（提取 → 转换 → 验证 → 加载）
- 本 SKILL 的四个 Phase 本身就是 Pipeline 模式的实践
