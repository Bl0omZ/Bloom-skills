# Inversion 模式

## 核心思想

Inversion（反转）模式：agent 先采访用户，再行动。
传统 agent 是「收到请求 → 立即执行」，Inversion 是「收到请求 → 先问清楚 → 再执行」。
减少猜测，提升输出相关性。

```
用户模糊请求
    │
    ▼
设计采访问题（按优先级排序）
    │
    ▼
提问（每次最多 3 个，避免用户疲劳）
    │
    ├── 用户回答完整 → 进入执行阶段
    ├── 用户部分回答 → 用默认值填充其余，说明假设
    └── 用户拒绝采访 → 用默认值全部填充，明确列出所有假设
```

## SKILL.md 模板

```markdown
---
name: [task]-assistant
description: |
  [任务] 助手，先了解需求再执行。当用户需要 [任务] 但描述不完整时使用。
  触发关键词：帮我做、我需要一个、定制化、[keyword]。
  流程：采访需求 → 确认理解 → 执行任务。
---

# [Task] Assistant

## Phase 1：需求采访

在执行任何任务之前，先提问以下问题。
每次最多提问 3 个，按优先级顺序提问。

P0 必问（影响任务方向）：
1. [最关键的问题]
2. [次关键问题]

P1 按需问（仅当 P0 答案不明确时）：
3. [细化问题]

P2 可选问（需要高质量输出时）：
4. [锦上添花的问题]

## Phase 2：确认理解

收集完答案后输出摘要，请用户确认：

  我的理解：
  - 任务：[摘要]
  - 约束：[摘要]
  - 期望输出：[摘要]
  - 假设（未确认项）：[列出所有默认值]

  是否正确？如有偏差请告知。

## Phase 3：执行

用户确认后执行任务。
若用户跳过确认直接说「开始」，视为确认。

## 边界处理

- 用户拒绝回答 → 使用合理默认值，在输出中明确标注「假设：...」
- 用户回答引入新歧义 → 最多追问 1 次，之后用最保守解释
- 采访超过 3 轮 → 停止追问，用当前信息执行，输出标注不确定项
```

## 采访问题设计原则

### 好问题 vs 坏问题

```
❌ 坏问题：你想要什么样的输出？（太宽泛）
✅ 好问题：输出给谁看？（开发者 / 最终用户 / 系统）

❌ 坏问题：有什么特殊要求吗？（邀请漫谈）
✅ 好问题：长度限制？（< 500字 / 500-2000字 / 不限）

❌ 坏问题：你期望的最终结果是什么？（重复任务描述）
✅ 好问题：成功的标准是什么？用什么衡量？
```

### 问题优先级排序规则

1. P0：影响「做什么」的问题（方向性）
2. P1：影响「怎么做」的问题（策略性）
3. P2：影响「做多好」的问题（质量性）

永远先问 P0，P0 答案清晰后再考虑是否需要 P1。

### 问题数量控制

- 最多 3 个必问问题（用户接受极限）
- 超过 3 个：合并相关问题，或改用选择题降低认知负担
- 选择题示例：「受众是？A) 开发者  B) 最终用户  C) 系统/API」

## 与其他模式组合

Inversion 通常作为其他模式的前置步骤：

- **Inversion + Generator**：采访收集变量 → Generator 填充模板
- **Inversion + Pipeline**：采访收集配置 → Pipeline 按步骤执行
- **Inversion + Tool Wrapper**：采访确定技术栈 → Tool Wrapper 注入对应文档

## 典型使用场景

- 用户请求模糊的创作任务（「帮我写篇文章」）
- 需要个性化定制的输出
- 多种实现路径，需用户选择方向
- 本 SKILL 的 Phase 1 本身就是 Inversion 模式的实践
