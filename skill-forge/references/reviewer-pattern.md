# Reviewer 模式

## 核心思想

Reviewer 按检查清单对输入进行评分，并按严重程度分级输出问题。
核心设计：检查清单驱动 + 严重程度分级 + 具体可操作的修复建议。

```
输入（代码/文档/设计/skill）
    │
    ▼
加载检查清单（references/checklist.md）
    │
    ▼
逐项检查 → 按严重程度分级
    │
    ▼
输出结构化报告（CRITICAL → HIGH → MEDIUM → LOW）
    │
    ▼
给出总体评分和优先修复建议
```

## SKILL.md 模板

```markdown
---
name: [domain]-reviewer
description: |
  [领域] 审查器。当用户需要检查/审查/评估 [对象] 时使用。
  触发关键词：审查、检查、评分、review、audit、质量评估、[domain]。
  输入：[对象]。输出：按严重程度分级的问题列表 + 总体评分。
---

# [Domain] Reviewer

加载 `references/[domain]-checklist.md`。

## 评分维度

按以下维度逐项评分（每项 0-[N] 分）：
1. [维度1]（[N]分）：[评分标准]
2. [维度2]（[N]分）：[评分标准]
3. [维度3]（[N]分）：[评分标准]

## 严重程度定义

- CRITICAL：必须立即修复，否则无法上线/使用
- HIGH：严重影响功能或安全，应在本次修复
- MEDIUM：影响质量，建议在本次或下次修复
- LOW：改善性建议，可选

## 输出格式

总体评分：X/10

### CRITICAL 问题
| # | 位置 | 问题描述 | 修复方案 |
|---|------|---------|----------|

### HIGH 问题
（同上格式）

### MEDIUM / LOW 问题
（同上格式）

### 总结
- 最优先修复：[1-2项]
- 整体评价：[1-2句]

## 边界处理

- 输入为空：要求用户提供待审查内容
- 输入超长（>2000行）：按模块分批审查，先审查最高风险模块
- 无问题发现：报告「未发现问题，评分 10/10」，列出检查的维度
- 超出领域范围：说明边界，建议使用其他 reviewer
```

## References 目录结构

```
references/
├── [domain]-checklist.md       # 完整检查清单（必须）
├── severity-examples.md        # 每个严重程度的真实示例
└── scoring-rubric.md           # 详细评分标准
```

## 检查清单文件格式

```markdown
# [Domain] 检查清单

## CRITICAL 检查项
- [ ] [检查项1]：[判断标准]
  - 触发条件：[什么情况下标记为 CRITICAL]
  - 示例：[真实反例]
- [ ] [检查项2]：...

## HIGH 检查项
- [ ] [检查项3]：...

## MEDIUM 检查项
...

## LOW 检查项
...
```

## Prompt 要点

- 严重程度定义必须具体，附真实示例（「SQL 注入」比「安全问题」好）
- 每个问题必须附带「修复方案」，不能只列问题
- 检查清单不超过 20 项（超过则按模块拆分为多个 reviewer）
- 总体评分要有明确的换算公式，不要主观估分
- 无问题时也要报告「已检查的维度」，证明确实做了检查

## 典型使用场景

- 代码安全审查（本 repo 的 code-audit SKILL）
- PR 代码质量检查
- SKILL.md 质量评估（本 SKILL 的 Phase 4）
- API 设计审查
- 文档完整性检查
- 架构方案评审
